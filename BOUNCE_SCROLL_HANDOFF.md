## 1. ゴール（要約）
1. **モバイル/タブレット（iOS・Android 問わず）でバウンスを実装**する。
2. **PC/デスクトップではバウンスさせない**。
3. バウンスで露出する領域の背景を、**その時のページ状態に応じた正しい色**で塗る。
4. **ホーム画面の上部**は、ヘッダーのグラデーションが **継ぎ目なく“本物のグラデ”として継続** して見えるようにする（本タスクの最重要ポイント）。

## 3. なぜこの設計になるのか（重要な技術背景）
- 弾性バウンス（ラバーバンド）が **ネイティブに存在するのは iOS と macOS のみ**。**Android・Windows・Linux・Firefox 大半はネイティブでは発生しない**（Android は「グロー/ストレッチ」表示のみ）。→ **Android でバウンスを出すには JS 実装が必須**。
- ネイティブ iOS バウンスは **露出域に固定背景しか出せず、グラデの動的継続は不可**（`::before` を `fixed` にしても実質ベタ塗り相当にしかならない）。
- 従って、**iOS/Android 双方で一貫したバウンス＋グラデ継続**を実現するには、両OSで **ネイティブバウンスを抑止（`overscroll-behavior`）し、JS制御のバウンスに統一** するのが妥当。JS制御なら **オーバースクロール量 `delta` を自分で把握** できるため、グラデをそれに連動させて継ぎ目なく継続させられる（= JS採用が今回のグラデ要望を可能にする）。
- **リスクと方針**: iOS ネイティブの慣性・弾性の“気持ちよさ”を JS で完全再現するのは難しい。**通常スクロールはネイティブのまま維持し、「境界でのオーバースクロールのみ」を JS で扱う方式を第一候補**とする（§6）。慣性フリングが境界に達したときの跳ね返りは既知の難所であり、**無理なら「フリング跳ねは無し（指ドラッグ時のみバウンス）」で妥協して良い**。感触は必ず実機で検証。

## 4. 対象プラットフォームと有効化条件
- **有効**: タッチ主体のスマホ/タブレット。判定は `matchMedia('(pointer: coarse)')` を主軸に、必要なら `navigator.maxTouchPoints` / UA（既存の `iPad|iPhone|iPod` 判定と整合）を併用。回転やモード変更で条件が動的に変わり得るので、**判定・適用は動的に**（既存の `resync`/リスナー方針と整合させる）。
- **無効**: デスクトップ（fine pointer）。Windows はネイティブでも出ないので実装不要。**macOS デスクトップのネイティブバウンスも `overscroll-behavior` で抑止**して「PC はバウンス無し」に統一する。
  - もし macOS デスクトップだけネイティブを残す方が良いと判断したら、勝手に決めず Nano に相談すること。
- タッチ判定の誤り（例: タッチ対応ノート PC が coarse を返す等）が無いか実機で確認すること。

## 5. 対象スクロール範囲（＝全領域）と各領域の露出色
以下の **全スクロール領域** に対応する。各領域は独立した内部スクロールを持つ。

| 領域 | スクロール実体 | 上端の露出色 | 下端の露出色 |
|---|---|---|---|
| ホーム（メイン） | body/ドキュメント（`phone-shell-inner` が伸長し body がスクロール） | **ヘッダーグラデの継続**（campus=緑 / station=紫、§7） | `--bg-page` |
| ドロワー（メニュー） | ドロワー本体内の `overflowY:auto` リスト領域 | リスト領域背景 `--bg-page`（上の固定ヘッダーは `--bg-card2`） | `--bg-page` |
| お知らせ / 設定 / ヘルプ | 各画面の `overflowY:auto` コンテンツ領域 | コンテンツ背景 `--bg-page`（上の NavBar 背景は **コードで確認** して整合） | `--bg-page` |
| PWA案内モーダル | カード内 `overflowY:auto` | カード背景 `--bg-card` | `--bg-card` |

- **色はハードコードせず**、既存の CSS 変数（`--bg-page` / `--bg-card` / `--bg-card2` / NavBar 実背景）と `.header.campus` / `.header.station` を **単一の真実源** として参照する（ドリフト防止）。
- **ダークモード**: 各変数はダーク色に自動追従。ヘッダーグラデは **テーマ非依存で不変**（現行 `.header.campus`/`.station` は `.dark` 下に別定義が無い）。ライト/ダーク両方で正しく見えること。
- ホーム下端と各オーバーレイは **単色でよい**（露出域＝各領域の背景色）。**グラデ継続が必要なのはホーム上端のみ**。

## 6. バウンス挙動の仕様（フィール）と実装アーキテクチャ

### 6.1 推奨アーキテクチャ（統一パターン）
各スクロール領域を「**背景を持つコンテナ** ＋ その中で `translateY` される **内側ラッパー**」として扱う。
- オーバースクロール時、内側ラッパーを減衰付き `translateY(delta)` で動かし、**空いた隙間からコンテナ背景が見える**ようにする。
- 各オーバーレイは「コンテナ背景＝その領域の露出色」を設定するだけで、**単色露出が自動的に成立**する。
- ホームのメインは、`phone-shell-inner` を内側ラッパー相当として動かし、**上端の隙間に §7 のグラデ背景レイヤー**を見せる（下端は body/キャンバス背景 `--bg-page` でよい）。

### 6.2 挙動
- 減衰（ラバーバンド）: 引くほど効きが鈍る逓減関数（例 `delta = maxPull * (1 - 1/(pull/c + 1))` 等）。最大オーバースクロール量に上限。
- リリースでスプリング/イージング（例 ease-out-quint 系）で `delta → 0` に復帰。
- 通常スクロールはネイティブ維持。**境界（`scrollTop<=0` で下方向、または最下端で上方向）でのみ** `touchmove`（`{ passive:false }`）で `preventDefault()` して JS バウンスに切り替える標準パターンを用いる。
- 各スクロール領域に `overscroll-behavior` を適切に設定し、**ネイティブのバウンス/グロー/スクロール連鎖/Android pull-to-refresh を抑止**（例: 領域に `overscroll-behavior: contain`／ルートは現行の `none` を維持）。その上で JS がバウンスを担う。iOS は 16+ 前提で可。
- **水平方向は無効**（`overscroll-behavior-x: none`、横スワイプナビ/水平バウンス無し）。
- **オーバーレイ表示中はメイン(body)のバウンス/スクロールを抑止**し、当該オーバーレイ領域のみ有効化（多重バウンス・背景貫通スクロール防止）。

### 6.3 実装の作法
- **再利用可能な単一のフック/ユーティリティ**（例 `useElasticOverscroll(scrollRef, { topFill, bottomFill, topGradient? })`）を作り、メイン＋各オーバーレイに適用する（重複実装を避ける、ミニマル方針）。
- **ライブラリ vs 自前**: 現状、web 向けに「Android にも弾性を足す」現役の定番ライブラリは乏しい（既存の elastic-scroll-polyfill / iNoBounce 等は Apple 限定や“抑止”用途で不適）。**プロジェクトのミニマル方針に沿い、軽量な自前実装を第一候補**とする。採用/不採用は最新の選択肢を調べた上で判断し、依存を足す場合は React18/TS5/PWA と整合するか確認、`package.json` / lock を適切に更新。
- デバッグ/診断用コードは検証後に除去（プロジェクト方針）。

## 7. ホーム上端のグラデ継続（最重要・詳細）
- **ゴール**: 上端オーバースクロールで露出する領域が、ヘッダーの `160deg` グラデの **継続** として見えること。ヘッダー上端(0%)と露出域の境界に **色段差・継ぎ目が一切出ない**。単なる別の色帯に見えてはならない。
- **なぜ可能か**: JS制御なので `delta`（引いた量）が分かる。露出域に敷くグラデ背景レイヤーを **`delta` に連動**させ、ヘッダーの幾何（幅・角度・色 stop）に合わせて **画素単位で連続** させる。
- **実装の指針（あなたが最良を選定・実機検証）**:
  - 例A: ヘッダーと **同一グラデを持つ“クッション”を上方向に延長** し、コンテンツごと `translateY(delta)` して露出させる（1要素1グラデで構造的に連続）。
    ※ `phone-shell-inner` は `overflow:hidden` なので、この延長分が **クリップされないレイヤー設計**（例: クリップ外の固定/兄弟レイヤーに描画し JS で位置を同期）にすること。
  - 例B: 上端に **固定のグラデ背景レイヤー** を置き、JS で高さ=`delta`・`background-position`/`background-size` をヘッダーに合わせて連続にする。
  - `160deg` の **斜め成分も含めて** 連続させることを目標に。
- **フォールバック（最後の手段）**: どうしても連動グラデが破綻する経路に限り、**上端色のベタ塗り**にフォールバック。
  - このグラデは上端が最も濃い（＝上方向は本来ほぼ一定色）なので、ベタ塗りでも継ぎ目自体は出ない。
  - ただし **まず本物のグラデ継続を目指す** こと。ベタ塗りは妥協策。
- **色の真実源**: `.header.campus` / `.header.station` を参照し、値を二重管理しない。
  - 現行値（要コード確認）: campus = `linear-gradient(160deg,#16a374 0%,#34d399 50%,#6ee7b7 100%)`、station = `linear-gradient(160deg,#818cf8 0%,#6c63d5 50%,#4f46e5 100%)`。
  - `route`（`campus_to_station` / `station_to_campus`）切替時に **色が正しく追従** すること。

## 8. 受け入れ基準（Acceptance Criteria）
- [ ] スマホ/タブレット（iOS Safari・iOS PWA standalone・Android Chrome・Android PWA）でバウンスが発生する。
- [ ] PC（Windows Chrome/Edge/Firefox・macOS Safari/Chrome）でバウンスが発生しない。色破綻・不要挙動も無い。
- [ ] **ホーム上端**バウンスでヘッダーグラデが **継ぎ目なく継続** して見える（campus/station 両方、ライト/ダーク両方）。route 切替で色が追従する。
- [ ] ホーム下端・各オーバーレイ（ドロワー・お知らせ・設定・ヘルプ・PWA案内）のバウンスで、各領域の背景色が正しく露出（`--bg-page` / `--bg-card` / `--bg-card2` 等）。ダークでも正しい。
- [ ] 縦/横/PC幅、iPad の Split View / Stage Manager などレイアウト変化でも塗りが正しく被る。
- [ ] 水平バウンス無し、Android pull-to-refresh 無し、スクロール連鎖無し。
- [ ] オーバーレイ表示中に背景メインが貫通スクロール/多重バウンスしない。
- [ ] 通常スクロールのなめらかさ・慣性が損なわれていない（実機体感）。
- [ ] §2 の不変条件を回帰させない。TypeScript strict を通す。
